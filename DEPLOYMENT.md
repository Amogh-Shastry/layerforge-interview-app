# Deployment

The app is a Next.js 16 (standalone output) app using Prisma + Neon Postgres and
OpenAI. It deploys to **Vercel** (native) and **Render** (Docker) with the same
codebase. Database is Neon — already migrated and seeded, shared by both.

## Environment variables (set these in each platform's dashboard)

| Var | Required | Notes |
|---|---|---|
| `DATABASE_URL` | ✅ | Neon **pooled** connection string, `?sslmode=require`. |
| `OPENAI_API_KEY` | ✅ | For voice + AI evaluation. Without it the app degrades to demo mode. |
| `AUTH_SECRET` | ✅ | Signs the HR session cookie. Generate: `openssl rand -base64 32`. (Falls back to `NEXTAUTH_SECRET` if unset.) |
| `HR_USERNAME` | ✅ | HR dashboard login. |
| `HR_PASSWORD` | ✅ | HR dashboard login. Use a strong value in prod. |
| `NODE_ENV` | auto/✅ | `production`. Vercel sets it automatically; the Dockerfile sets it for Render. |

> These are **never** committed (`.env*` is git-ignored and dockerignored). Set them
> in the platform dashboards. Rotate any secret that has been shared in plaintext.

---

## Vercel

1. Import the GitHub repo at vercel.com → New Project. It auto-detects Next.js.
2. Build command stays the default `npm run build` — it runs `prisma generate && next build`
   (the Prisma client is generated at build time; required on Vercel).
3. Add the env vars from the table above (Project → Settings → Environment Variables).
4. Deploy. The `proxy.ts` (Next 16 middleware) runs on Vercel's Edge automatically.

No `vercel.json` is needed. `output: "standalone"` is harmless on Vercel.

---

## Render (Docker)

Option A — Blueprint (recommended):
1. Render → New → Blueprint, connect this repo. It reads [`render.yaml`](render.yaml).
2. Fill in the `sync: false` secrets (`DATABASE_URL`, `OPENAI_API_KEY`, `HR_USERNAME`,
   `HR_PASSWORD`). `AUTH_SECRET` is auto-generated by Render.
3. Create → it builds the Dockerfile and deploys.

Option B — Manual: New → Web Service → Docker → point at this repo. Health check path `/`.
Add the env vars from the table. Render injects `PORT`; the standalone server binds to it.

The Dockerfile builds the standalone output and runs `node server.js` as a non-root user.

---

## Database

Both platforms point at the same Neon `DATABASE_URL`, which is already migrated and
seeded. If you ever target a **fresh** database, run once from your machine with that
DB's URL in `.env.local`:

```
npm run db:push   # create tables
npm run db:seed   # org + HR user + 3 job positions
```

(The production images don't include the Prisma CLI, so run migrations from local or CI,
not from the running container.)
